引言

当单智能体 RL 扩展到多智能体场景时，环境从 MDP 变为随机博弈 (Stochastic Game)。每个智能体的最优策略依赖于其他智能体的行为，导致非平稳性 (non-stationarity) 成为核心挑战。本文从系统架构视角剖析 MARL 的设计空间。

2.1 值分解方法

QMIX (Rashid et al., 2018) 将联合动作值函数分解为单调混合：

Qtot(τ,a;s)=fmix(Q1(τ1,a1),…,Qn(τn,an);s)Q_{tot}(\boldsymbol{\tau}, \mathbf{a}; s) = f_{\text{mix}}\left(Q_1(\tau_1, a_1), \ldots, Q_n(\tau_n, a_n); s\right)Qtot​(τ,a;s)=fmix​(Q1​(τ1​,a1​),…,Qn​(τn​,an​);s)

约束条件确保 IGM (Individual-Global-Max) 一致性：

5. 代码骨架：基于 PyTorch 的 QMIX

class QMIXMixer(nn.Module):
    def __init__(self, n_agents, state_dim, mixing_dim=32):
        super().__init__()
        self.hyper_w1 = nn.Sequential(
            nn.Linear(state_dim, mixing_dim),
            nn.ReLU(),
            nn.Linear(mixing_dim, n_agents * mixing_dim),
        )
        self.hyper_w2 = nn.Sequential(
            nn.Linear(state_dim, mixing_dim),
            nn.ReLU(),
            nn.Linear(mixing_dim, mixing_dim),
        )
        self.hyper_b1 = nn.Linear(state_dim, mixing_dim)
        self.hyper_b2 = nn.Sequential(
            nn.Linear(state_dim, mixing_dim),
            nn.ReLU(),
            nn.Linear(mixing_dim, 1),
        )

    def forward(self, q_values, state):
        # q_values: (batch, n_agents)
        # state: (batch, state_dim)
        batch_size = q_values.size(0)
        w1 = torch.abs(self.hyper_w1(state)).view(batch_size, self.n_agents, -1)
        b1 = self.hyper_b1(state).view(batch_size, 1, -1)
        w2 = torch.abs(self.hyper_w2(state)).view(batch_size, -1, 1)
        b2 = self.hyper_b2(state).view(batch_size, 1, 1)

        x = torch.bmm(q_values.unsqueeze(1), w1) + b1
        x = torch.relu(x)
        q_tot = torch.bmm(x, w2) + b2
        return q_tot.squeeze(-1).squeeze(-1)

总结

MARL 系统设计的核心权衡在于信息共享程度 vs. 可扩展性。CTDE 在训练效率和部署可行性之间取得了良好平衡。未来方向包括：基于 LLM 的智能体通信、离线 MARL、以及 sim-to-real 的多机器人迁移。
